## 🖥️ Remote Desktop / Shared VPS Architecture

### The Need
Marcus and Darien need a **shared business operating system** — a virtual desktop they can both log into from anywhere that serves as:
- Central workspace for all business activities
- Testing environment for AI integrations
- Project management hub
- Secure storage for credentials and documents
- Collaboration space for both Vespera and Torchline work

### Recommended Stack: Ubuntu + NoMachine + Tailscale VPN

```
│  └── 📦 Docker services:                               │
│      ├── n8n                                           │
│      ├── Budibase                                      │
│      ├── Ariella (bot)                                 │
│      └── Supabase (self-hosted)                        │
│                                                         │
│  Storage: 100GB SSD + DigitalOcean Volumes (expandable)│
└─────────────────────────────────────────────────────────┘
```

### Access Methods
- **NoMachine client** — Native desktop experience (fastest, best performance)
- **Browser** — NoMachine web player (no install needed)
- **Tailscale** — Secure network access to all services
- **SSH** — Terminal access for advanced users

### Why NoMachine over RDP/VNC?
- ✅ Better performance (NX protocol)
- ✅ Audio forwarding
- ✅ File sharing via drag-and-drop
- ✅ Works through NAT/firewall
- ✅ Cross-platform clients (Win/Mac/Linux)
- ✅ Free for personal use (4 connections)

### Why Ubuntu vs Windows Server?
- ✅ Free (no licensing cost)
- ✅ Better for Docker/containers
- ✅ Better for AI/ML workloads
- ✅ Lower resource usage
- ✅ Better for headless deployment
- ❌ Less familiar for Windows users (but NoMachine makes it feel native)

### Our Model Selection Strategy
1. **Task-specific routing** — Route each task to the best/cheapest model for that task
2. **Cost optimization** — Use cheap models (MiniMax, Qwen) for 80% of tasks
3. **Quality gates** — Use expensive models (Claude Opus, o3) for critical code review
4. **Self-hosting** — Run Qwen 3.6-35B locally for sensitive/uncensored tasks
5. **Redundancy** — Always have 2 models per task type in case one is down
